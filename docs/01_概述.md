## 文档规范

- 采用 **Markdown** 格式，UTF-8 编码（LF 换行）
- 中文为主，技术术语保留英文，关键概念标注中英对照
- 代码块标注语言类型
- 架构图使用 PNG 存放于 `res/` 目录
- 文档命名：带序号中文命名，如 `01_概述.md`、`02_架构设计.md`，子目录下为 `01_开发指南.md`

## 设计文档编写规范

在对新功能、Bug 修复或重构进行设计时，应编写设计文档，存放于 `docs/designs/` 目录下。

### 目录结构

```
docs/designs/
├── features/       # 新功能设计
├── refactors/      # 重构设计
└── bugfixes/       # Bug 修复设计
```

每个主题按 kebab-case 创建子目录，同一主题的迭代版本用新日期文件：

```
docs/designs/features/im-conversation-sync/
├── 2026-03-19-im-conversation-sync-design.md
└── 2026-04-21-im-conversation-sync-design.md
```

### 分类规则

| 类别 | 目录 | 适用场景 |
|------|------|----------|
| 功能 | `features/` | 新增用户可感知的能力 |
| 重构 | `refactors/` | 不改变外部行为的内部改造 |
| Bug 修复 | `bugfixes/` | 修复已有功能的异常行为 |

### 文件命名

```
YYYY-MM-DD-<主题描述>.md
```

- 日期为文档创建日期，主题描述使用 kebab-case
- 同一主题多次迭代，每次新建带新日期的文件，旧版保留作历史参考

### 文档语言

中文。

```markdown
### 1.1 问题/动机
### 1.2 目标

## 2. 现状分析

## 3. 方案设计

## 6. 验证计划
```

### 编写原则

1. **一个文件一个完整文档** — 所有内容合并在一个文件中
2. **设计迭代用新文件** — 同一主题的新版本，新建带新日期的文件，旧版保留作历史参考
3. **自包含** — 每个文件应独立可读，不强依赖其他文件才能理解
4. **重实质轻形式** — 上述章节结构为参考，根据实际内容灵活调整
